# reflecmino評価関数

## generator.jl

generator.jlにある`generate`関数がパズルの盤面を生成します。 \
`generate(mode::Int, seed::Int)::PuzzleData`では`mode`と`seed`を入れてください。

`mode`は1,2,3のいずれかで、1が普通のReflecMino、2がReflec鬼Mino、3がReflec鬼Minoの裏ステージに対応しています。 \
`seed`は32bitに収まる整数でお願いします。本家ReflecMinoとReflec鬼Minoでは日付を`seed`として与えています。(例: 2026年6月27日 → seed=20260627) \
generator.jlの439行目付近に`DIFFICULTY`という定数があります。これも難易度に対応していて、1,2,3のいずれかに設定しておいてください。

`generate(seed::Int)::PuzzleData`は`generate(DIFFICULTY,seed)`と等価です。

---

`PuzzleData`型はそのままでは使えないので`minos_start_end`関数に通してください。 \
`minos_start_end(puzzle::PuzzleData)::Tuple{Vector{Tuple{CellData, CellData, CellData}}, Tuple{Point, Point}, Tuple{Point, Point}}`関数は`PuzzleData`型の変数を引数にとり、
ミノの情報とスタートとゴールの位置を返します。 \
`CellData`型は以下のように定義されています。

```julia
struct CellData
    x::Int
    y::Int
    type::Int8
end
```

`minos_start_end`関数から得られる`Vector{Tuple{CellData, CellData, CellData}}`の部分のデータは、一つ一つのミノの3つのマスの情報になっています。

### `CelData`の`type`について

|中身|表す情報|
|----|----|
|Int8(0)|空白マス|
|Int8(1)|壁|
|Int8(2)|空白ピースマス|
|Int8(3)|slash '/'|
|Int8(4)|back slash '\\'|

という対応があります。
`x`と`y`には原点をどこか固定したときの各ミノのマスの座標が入っています。 \
例えば、CellData(0, 0, 4), CellData(-1, 0, 2), CellData(0, 1, 3)なら \
(0,0)の位置に4(つまり'\' 右肩下がりに置いた両面鏡) \
(-1,0)の位置に2(つまり空白マス) \
(0,1)の位置に3(つまり'/' 右肩上がりに置いた両面鏡) \
ということを表し、このピースは逆さL字型のピースであることが分かります。 \
→がx軸の正の方向で、↓がy軸の正の方向です。

`minos_start_end`関数から得られる`Tuple{Point, Point}, Tuple{Point, Point}`の部分のデータは、前者がレーザーの`start`の位置、後者がレーザーが入るべき`end`の位置を表しています。 \
5×5の盤面の左上を`(1,1)`とし、右下を`(5,5)`とする座標を取っています。レーザーの`start`と`end`は壁に埋まっているので、`x`座標と`y`座標のどちらかが、`0`もしくは`6`の値を取ります。 \
`Point`構造体の定義は以下のようになっています。

```julia
struct Point
    x::Int
    y::Int
end
```

## rating.jl

`rating.jl`にある`evaluate_s(target_seed::Integer)`関数が1ステージに対する評価関数となります。引数は32bitに収まる整数値を入力してください。

内部で使用している評価関数の評価方法について解説します。

### 1. 各盤面（状態 $s$）の罠度 $f(s)$ の計算

目指すのは、「見た目はクリア寸前（レーザーは大満足）なのに、中身は正解から遥かに遠い（ミノの配置はデタラメ）」という、人間が「解けた！」と錯覚して抜け出せなくなるローカルミニマ（罠盤面）に高いスコアを付与する関数です。

#### Step A: 配置的近さ（操作的距離）の算出：$Prox_{\text{mino}}(s)$

正解状態（`true_state`）から、逆BFSによって計算された物理的な本物の最短手数（`distances[i]`）をベースに、盤面の近さを $0.0 \sim 1.0$ に正規化します。

$$Prox_{\text{mino}}(s) = \frac{1}{1 + \text{distances}[i]}$$

* ミノの配置が完全に正解（手数が $0$）のとき、配置的近さは $1.0$（最大値）。
* 正解から手数が離れるほど、配置的近さは $0.0$ に近づく。
* ※物理的に繋がっていない（逆BFSの波が届かなかった）状態は、人間が迷い込む余地がないため、評価対象から除外（$f(s) = 0$）されます。

#### Step B: 機能的近さ（レーザー評価）の算出：$Prox_{\text{laser}}(s)$

現在の盤面のレーザースコア（`laser_score`）が、正解状態の最大スコアにどれだけ肉薄しているかを評価します。

$$Prox_{\text{laser}}(s) = \frac{\text{scores}[i].\text{laser score}}{\text{max laser score}}$$

* レーザーが正解ルートを完璧に捉えているとき、機能的近さは $1.0$ になります。
* 最後の1枚の鏡の向きだけが違っていて、途中まで完璧に光が流れている「おあつらえ向きの罠盤面」では、この値は $0.9$ などの非常に高い値になります。

#### Step C: 盤面 $s$ の罠度（ローカルミニマ度）の決定：$f(s)$

上記2つの「近さ」を組み合わせ、人間の錯覚度を数値化します。

$$f(s) = Prox_{\text{laser}}(s) \times (1.0 - Prox_{\text{mino}}(s))$$

* 光は綺麗に流れている（$Prox_{\text{laser}} \to 1$）のに、手数は正解から絶望的に遠い（$Prox_{\text{mino}} \to 0$）とき、この $f(s)$ は $1.0$ に極めて近くなり、「最強の罠盤面」として定義されます。
* 逆に、完全に正解の盤面では $Prox_{\text{mino}} = 1.0$ となるため、$(1.0 - 1.0) = 0$ となり、罠度は必ず $0$ になります。

---

### 2. 全状態の滑らかな足し上げと最終難易度 $D_{\text{stage}}$ の統合

ステージ全体の難易度を決める際、特定の閾値で状態を足し切り（カウント）にすると、「スコア 0.69 の罠は無視され、0.71 の罠は 1 と数えられる」といった不連続性が生まれてしまいます。

本数理モデルでは、統計物理学の分配関数に着想を得た「エネルギー局所和（ソフトマックス型加算）」を採用し、全状態の $f(s)$ を滑らかに足し上げています。

#### 統合数式

$$D_{\text{stage}} = \frac{1}{\alpha} \log \left( \sum_{i=1}^{N} \exp(\alpha \cdot f(s_i)) \right)$$

### この数式の特性

* 感度パラメータ $\alpha$（コード内では `10.0`）： \
$\alpha$ を大きく設定することで、無害な低スコア状態（$f(s) \approx 0.1$）の雑音は指数関数的に消し去られ、本物の極悪な罠（$f(s) \ge 0.8$）だけが強烈に増幅されて難易度に寄与します。
* 「罠の鋭さ」と「沼の広さ」の同時評価： \
深くて鋭い罠が1つあるだけでもスコアは跳ね上がり（$D_{\text{max}}$ の性質）、同時に、そこそこの罠が広範囲に広がっている（$D_{\text{volume}}$ の性質）場合も綺麗に合算されます。
* 対数（$\log$）によるスケール調整： \
最後に $\log$ を取ることで、状態数が数万〜数十万件に爆発しても、難易度スコアを人間が直感的に比較しやすい現実的なスケール（通常 $0 \sim 15$ 程度）に収束させています。

## analyze.jl

`analyze.jl` には、`rating.jl` で計算したデータを年単位で収集する関数、収集した巨大な生データをディスク容量を圧迫しないように軽量化して保存・展開する関数、そして蓄積されたデータからパズルのマクロな傾向をあぶり出す分析関数群が含まれています。

### 1. データ構造（構造体）の定義

効率的なデータ管理と軽量化保存のために、2つの構造体が定義されています。

#### `StageEvaluationResult`

1つのステージの評価後に得られる「すべての生データ」を保持する構造体です。全状態の探索結果が丸ごと入るためデータサイズは非常に大きくなります。

```julia
struct StageEvaluationResult
    D_stage::Float64               # ステージの最終難易度スコア
    distances::Vector{Int}         # 全状態から正解までの最短手数の配列
    correct_states::Vector{UInt64} # 正解（ゴール）となる状態のビット配列
    scores::Vector{EvaluatedState} # 全状態の個別レーザースコア等の配列
end

```

#### `LightStageResult`

生データから「統計・分析に必要な情報」だけを抽出した、保存専用の軽量構造体です。巨大な配列（`distances` や `scores`）をすべて排除することで、データサイズを数万分の一に圧縮します。

```julia
struct LightStageResult
    D_stage::Float64               # ステージの最終難易度スコア
    num_nodes::Int                 # 総状態数（ノード数）
    num_corrects::Int              # 正解状態数（別解の数）
    max_distance::Int              # 正解ステージへの最大手数（パズルの深さ）
    f_scores_summary::Vector{Float64} # 分布サンプリング用（通常は空配列）
end

```

---

### 2. データ収集と軽量化保存の関数

#### `collect_one_year_data(year::Int)::Dict{Int, StageEvaluationResult}`

指定された年（`year`）の1月1日から12月31日までの365日分のシードをループで回し、`evaluate_s` を実行して結果を辞書型（`Dict`）にまとめて返します。

* 1件あたり約800ms、1年分（365件）の処理を約5分で完了します。

#### `save_year_data_light(year_data, filepath)` / `(year_data, year::Int)`

`collect_one_year_data` で得られた重い生データから、必要な統計量（状態数、正解数、最大手数）だけをその場で計算し、`LightStageResult` に詰め替えてバイナリ形式（`.jls`）で保存します。

* 1年分のデータが数KB〜数十KBまで圧縮されます。
* 引数に `year`（整数）を直接渡すと、`puzzle_year_datas/puzzle_year_data_$(year).jls` という規則的なファイル名で自動保存されます。

#### `load_year_data_light(filepath)::Dict{Int, LightStageResult}`

軽量保存された `.jls` ファイルを指定してディスクから読み込み、分析用の辞書データとして復元します。

---

### 3. 難易度解析レポート関数

蓄積されたデータセットを読み込み、数理モデルの健全性やパズルの傾向を人間の目に見える形でレポート出力する2つの関数です。

#### `analyze_year_data_light(light_data::Dict{Int, LightStageResult})`

特定の1年分の軽量データを対象に、詳細な統計レポートをコンソールに出力します。

#### `analyze_historical_puzzle_data(base_dir::String="puzzle_year_datas")`

指定されたディレクトリ内(このレポジトリでは`puzzle_year_data`フォルダの中)にある1900年から2026年までの127年分（約46,000日分）の軽量データを一挙にロードし、一括分析を行います。

### レポートで出力される主な分析項目

1. 全体難易度統計: \
全期間の難易度の平均値、中央値、ばらつき（標準偏差）に加え、歴史上最も難しかった日と最も簡単だった日を特定します。
2. 難易度帯の分布（ティア）: \
スコアの平均（$\mu$）と標準偏差（$\sigma$）を基準に、各ステージを「鬼 (Oni)」「難 (Hard)」「普通 (Normal)」「易 (Easy)」の4段階に自動で分類し、その割合を算出します。 \
(※ Oniという名前はReflec鬼Minoとは関係ないです。`DIFFICULTY=1`でもOni,Hard,Normal,Easyの評価で分けます。)
3. 変数間の相関係数（ピアソン相関）:

* 総状態数（ノード数）との相関 : 盤面が広いステージほど本当に難しくなっているかを検証します。
* 正解状態数（別解の数）との相関 : クリアの選択肢が多いほどステージが簡単になるか（負の相関があるか）を検証します。
* 正解への最大手数との相関 : ステージの「深さ」が人間の感じる難しさにどれだけ寄与しているかを証明します。

1. 月別平均難易度の推移: \
1月〜12月の月ごとの平均難易度をテキストバーチャートで視覚化し、カレンダーアルゴリズムによるシード値の生成に偏りがないかをチェックします。
1. 歴史的難関トップ5: \
127年間の全パズルの中で、難易度スコアが高かった上位5ステージのシード値、状態数、最大手数をランキング形式で一覧表示します。

\
※ `analyze_historical_puzzle_data` の戻り値は、各統計配列（`seeds`, `d_stages`, `num_nodes`, `num_corrects`, `max_dists`）がフラットに並んだ名前付きタプルとなっており、 \
そのまま `Plots` 等のライブラリに渡してヒストグラムや散布図として可視化することができます。
